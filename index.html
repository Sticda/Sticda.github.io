Bean 的生命周期

BeanPostProcessor接口里面的postProcessBeforeInitialization（还没有开始初始化）-> InitializingBean里面的afterPropertiesSet（属性被设置之后）->init-method(bean的声明定义中) -> BeanPostProcessor接口里面的postProcessBeforeInitialization(初始化完成）-》用完之后-》DisposableBean里面的destory方法-》destory-method（Bean 的声明定义中指明的自定义方法）

ComponentScan

spring自动扫描和这个注解在同一个包下所有带了@Component的类，创建Bean，@component和@controller、service、repository一样，componentscan里面可以加basePackages（string数组是包名）和BasePackageClasses（类或者接口所在的包），没有匹配的bean时会抛出异常，@Autowired可以设置required属性为false避免抛出异常。

Java Config

@Configuration类里的@Bean注解告诉spring返回一个bean，bean id默认是方法名，@Bean修饰的方法并不会每一次返回不同的实例，默认实例是单例模式，只要是调用过一次，就会拦截再次调用，直接返回之前的实例。

环境与profile

@Profile会指定哪个Bean属于哪一个profile，可以修饰方法和类，在xml中beans元素的属性profile可以指定，xml中beans下面可以嵌套定义beans，spring.profile.active和spring.profile.default表明激活的profile是哪个，在测试环境中使用@ActiveProfile表明启动哪个profile

条件化Bean

@Conditional注解的括号里是实现了condition接口的类，通过这个类得到一个boolean，如果是true的话就创建Bean，false就不，这个condition接口实现的方法有两个参数，一个是ConditionContext，可以获得很多关于应用的信息，另一个是AnnotatedTypeMetadata，可以获得同一类下面的其他注解。@Profile就是一个@conditional类型的注解

限定自动装配的Bean

@Qualifier是可以限制选择谁作为装配的Bean，括号里面是bean id的字符串，但是当某个类被重构之后，bean id就会发生变化，可以通过@Qualifier为某个类指定特殊的限定符，不使用bean id作为限定符，这样就不害怕某个类的类名发生变化，但是还有一种可能就是又会出现重复，所以可以自定义注解，通过继承@Qualifier就可以实现限定，将自定义注解标注到需要注入的地方和类定义的地方，就可以保证唯一的注入。

  1. 同时有多个bean满足自动装配的条件的话可以利用@Primary指定首要加载。
  2. 所有使用@Component注解声明的类都会创建为bean，并且id是首字母变为小写的类名。
  3. c命名空间和正常版的区别：正常版(constructor-arg)可以注入列表，但是c-命名空间不可以
  4. @Import导入java config，@ImportResource导入xml文件
  5. classpath 路径在每个J2ee项目中都会用到，即WEB-INF下面的classes目录，所有src目录下面的java、xml、properties等文件编译后都会在此。
  6. @Scope作用域有四个：prototype是指每次注入或者每次通过spring应用上下文获取的时候都是不同的，ConfigurableBeanFactory和WebApplicationContext下面的常量是scope的value，scope 的另外一个参数proxyMode属性解决了将会话或者请求bean注入到单例bean里的问题，spring会注入一个代理类，然后单例类真正调用其他类的时候，才会去把调用委托给真实存在的类，ScopedProxyMode.INTERFACES(抽象类或者接口) ;ScopedProxyMode.TARGET_CLASS(如果是具体的类，必须使用CGLib生成基于类的代理)

20181101-hadoop

2018-11-01

Hadoop

主要内容

  1. Common：常用工具
  2. Avro：用于数据序列化的系统
  3. MapReduce：MapReduce中的Map可以将输入的键值对映射成新的键值对，然后Reduce会对所有相同key的所有value进行处理再输出键值对作为结果
  4. HDFS：hadoop distributed file system 高容错率
  5. Chukwa：开源数据收集系统
  6. Hive：数据仓库，可以通过他自己的查询语言（类似sql）查询hadoop中的数据
  7. HBase：分布式，面向列的开源数据库。表是疏松的，每一行可以有不同的列
  8. pig：对数据集进行评估和分析
  9. ZooKeeper:为分布式应用设计的开源协调服务

HDFS 体系结构

由一个NameNode和若干个DataNode组成，NameNode作为主服务器，负责管理文件系统的命名空间和外部设备对于文件的访问，而DataNode则是管理具体存储的数据，一个文件被分成若干数据块放在一些DataNode上，NameNode负责管理文件系统的命名空间，例如打开关闭重命名文件等，管理数据块和DataNode的映射关系，DataNode负责处理具体的文件读写请求，在NameNode的调度下完成数据块的具体操作，也就是DataNode对于数据块的操作需要受到NameNode的管理，用户需要保存的数据不会经过NameNode，直接流向存储数据。

MapReduce 体系结构

一个单独运行在主节点的JobTracker和一些运行在从节点上的TaskTracker，主节点负责分发任务和配置信息，并重新执行失败的子任务，从节点就是执行主节点分发的任务。

一个输入的键值对经过用户自己定义的Map函数，产生中间键值对，有一个迭代器会把具有相同键的中间键值对交给用户自定义的Reduce函数，产生最后的输出。一般来说，Reduce函数的输出是0或者1。

JobTracker可以运行在集群中的任意一个节点上，但是TaskTracker只能运行在DataNode上

数据分割（Partition）：将map任务输出的中间结果按照key的范围划分为R份，保证某一段范围的key是由一个reduce函数处理的。

在Hadoop上运行的作业需要指明输入和输出，并通过实现合适的接口或者抽象类提供Map和Reduce函数，同时指定参数构成作业配置，JobTracker会分发作业和配置信息给TaskTracker。

Hadoop 数据管理

分布式文件系统HDFS、分布式数据库Hbase和数据仓库工具Hive

HDFS

NameNode主要负责文件系统的命名空间、集群配置信息和存储块的复制等，NameNode会将文件系统的metadata存储在内存中，这些信息包括了文件信息，每个文件对应的文件块信息以及每一个文件块在DataNode中的信息，DataNode将文件块存储在本地文件系统，保存了所有block的metadata，同时周期性的发给NameNode，client就是需要获取文件的应用程序。

  1. 文件写入：client向NameNode发起写入文件的请求，然后NameNode会根据文件大小和文件块的信息，返回给client一堆DataNode的信息，client将文件划分为block，存入到DataNode里
  2. 文件读取：client向NameNode发起读取的请求，NameNode返回给client文件对应的DataNode的信息，然后client读取。
  3. 文件备份：文件有最小复制数，一般为3，也就是说这个文件在整个系统中存在3份，一份是存放在NameNode指定的DataNode上，一份存在与指定DataNode不在一台机器上的DataNode，最后一份存在与DataNode在同一个机架上的DataNode。
  4. 数据管道性的写入：文件写入的时候是一个一个block的写入，先写一个block，然后备份这个block，等这个block备份完成的时候才写下一个block
  5. 安全模式：刚开始启动的时候不允许文件的写入和读取，直到安全模式结束

HBase的数据管理

HBase类似于Bigtable，稀疏的、多维度的排序映射表，表的索引有行关键字、列关键字、时间戳。数据都是字符串，没有类型，每一行可以有不同的列，列名字的格式是“:

由三个部分组成

HBaseMaster：主服务器

一个HBase只部署一台主服务器，通过领导选举算法确保只有唯一的主服务器是活跃的，zookeeper中保留了主服务器的信息，如果主服务器瘫痪，则会利用领导选举算法从备用中选择新的。主服务器负责集群的初始化，从hdfs中获取根域目录。负责集群中域的分配、域服务器的监视

HRegionServer：域服务器

服务于主服务器分配的域，处理客户端读写请求，本地缓冲回写、本地数据压缩等，每个域只有一台域服务器，开始运行的时候会从HDFS中读取域的日志和所有的存储文件，同时管理HDFS文件的持久性存储工作

HBase Client：客户端

查找用户域所在的域服务器，从根域到元域，再到用户域所在的域服务器，之后这个地址将会被客户端缓存。

Hive的数据管理

Hive是建立在Hadoop上的数据仓库基础框架，数据提取、转化、加载，Hive QL类似于SQL，按照使用层次可以分为元数据存储、数据存储和数据交换

元数据存储：

元数据存在RDBMS中，三种模式可以连接到数据库：single user mode（连接到一个In-memory的数据库Derby）multi user mode（网络连接到数据库） remote server mode（非java客户端访问元数据库，在服务端启动一个metaStoreServer，客户端利用thrift协议通过MetaStoreServer访问数据库）

数据存储

Hive没有专门的数据存储格式，创建表的时候需要让Hive知道列分隔符和行分隔符，Hive中所有的数据都存储在HDFS中，有四种数据模型：Table、External Table、Partition和Bucket

数据交换

用户接口主要有client客户端、web界面以及数据库接口，hive将元数据存储在关系数据库中，解释器、编译器、优化器、执行器会负责HiveQL 的编译和执行，Hive的数据存储在HDFS中。

MapReduce

Map函数会接受一个<key,value>的形式的输入，并且产生同样形式的中间输出，Hadoop会将所有具有相同key的value集合在一起传递给Reduce函数，Reduce接受一个<key,list of values>形式的输入，然后对这个value集合进行处理并且输出结果

#Java的语法糖

  1. 泛型和类型擦除：java中的泛型只存在于源代码中，编译之后的字节码就会类型擦除把泛型擦掉，然后在使用泛型的地方使用强制类型转换。
  2. 自动拆装箱：基本数据类型不是对象，通过自动拆装箱互相转换，内部还是通过函数调用实现
  3. 变长参数：变长参数具有相同类型，位于参数列表的最后，内部实现是java数组
  4. 增强for循环：可以轮询一个数组或者实现了iterable接口的对象。内部使用iterable接口进行遍历
  5. 内部类：定义在一个类内部的类，编译器还是会为内部类生成一个单独的class文件
  6. 枚举类：在Java的类型中并没有enum类型，枚举类会被编译成普通类，继承java.lang.enum，并被final关键词修饰。

  2. 乘法和除法可以使用移位操作，乘除2的n次方，可以左移或者右移n
  3. 循环内不要不断创建对象引用

  19. 一个用final定义的变量，不管它是类型的变量，只要用final定义了并同时指定了初始值，并且这个初始值是在编译时就被确定下来的，那么这个final变量就是一个宏变量。编译器会把程序所有用到该变量的地方直接替换成该变量的值，也就是说编译器能对宏变量进行宏替换。

2018-08-13
  1. System.out.println的具体执行方法里加了Synchronized同步块。
  2. 守护线程：守护用户线程，在用户线程结束后自动结束，thread的setDaemon（true）必须在start之前执行，不能将正在运行的用户线程设置为守护线程。在守护线程里产生的线程也是守护线程
  3. ThreadLocal:线程本地变量，每个线程都会在本地存储一个副本，不会对其他线程的变量副本造成影响，ThreadLocal可以重写的方法包括initialValue、ge、set、remove，ThreadLocal底层是由ThreadLocalMap实现的，每个Thread都有自己的一个ThreadLocalMap，为了避免内存泄漏，key都是对ThreadLocal 的弱引用，value是变量的本地值，防止内存泄漏的方法：ThreadLocal已经被回收，但是ThreadLocalMap中仍然被强引用，会有内存泄漏。replaceStaleEntry会将所有key为null 的value设置为null，expungeStaleEntry方法将键值都为null的Entry设置为null。ThreadLocal的使用场景是每个线程都有自己的变量副本，还有就是实例需要在多个方法之间共享，但却不想在多个线程之间共享。
  4. CountDownLatch倒计时器：CountDownLatch有四个方法，构造器CountDownLatch（int n）、await、await（time,unit），countdown，等待n个线程执行完毕之后，才会继续接下来的操作，每个线程执行完之后都要调用countdown函数使得倒计时器减1，在主线程中调用await方法，就会使得主线程阻塞直到倒计时器数为0，才会继续执行操作。
  5. CycleBarrier循环栅栏：所有的线程需要等待到达一个状态之后才可以继续进行下面的操作，CycleBarrier有两个参数是n和Runnable barrierAction，n就是当CycleBarrier上等待的线程到达n时就会继续下面的操作，CycleBarrier也有一个await方法，使得当前线程暂停等待，在线程执行完之后，还可以调用Runnable的action继续后续的操作，也可以没有barrierAction这个参数。
  6. Semaphore信号量：信号量可以控制访问变量的线程个数，通过acquire获取许可，release释放许可（acquire和release都是阻塞的）还可以释放或者获取多个许可，前面加try就是非阻塞的，构造器参数就是permits或者permits和fair（等待越久越先获得许可）。
    7. 线程的终结：线程有一个stop方法，调用它可以直接终止线程，但是这个方法是Deprecated（过时的），因为直接stop线程会导致正在进行的线程发生数据不一致的问题，方法应该是在循环中检验一个变量的值，完成业务之后修改变量的值，导致循环退出，这样在循环的一个流程里面执行完业务，然后退出。
  7. sleep和wait的区别：sleep是让线程休眠，到时间之后会自动唤醒，然而wait是等待，必须等待唤醒才可以使用。sleep是Thread线程类的静态方法，而wait是Object顶级类的方法，sleep可以在任何地方使用，但是wait只能在同步代码块中使用，而且调用wait的类就是同步代码块锁定的类；sleep和wait的区别还在于sleep在休眠的时候并不会释放已经获得的锁，而wait就会释放已经获得的锁，sleep和wait都会暂停当前线程让出CPU执行时间；sleep需要捕获异常（在休眠的时候被异常中断），而wait不需要。
  8. JVM重排序规则：volatile和监视器锁；final字段（1.final字段的写入（构造器内）以及final字段对象本身的引用写入都不能和后续的构造器外持有该final字段的对象的写入重排序，2.final字段的第一次加载不能和持有这个final字段的对象的写入重排序）；内存屏障（LoadLoad,StoreStore,LoadStore,StoreLoad,最后这个最常用，先store后load不能打乱，防止读到旧数据，但是StoreLoad开销很大）；HappenBefore规则
  9. 内存泄漏：有一些对象已经不需要使用了，按理说这部分内存应该被垃圾收集器回收，但是这部分对象仍然在被引用，所以无法回收，程序也无法使用，导致内存泄漏。防止内存泄漏：注意集合类ArrayList、HashMap等，因为他们是内存泄漏常见的地方，注意事件监听器和回调，如果一个类自己管理自己的内存，那么指向其他对象的成员变量需要为null。
  10. Java6中字符串的subString方法存在内存泄漏的问题：因为JDK6中的string底层是char数组和起始标识以及数组长度，substring只会修改新生成字符串的起始标识和数组长度，可能会导致原来存在的一个很长的字符串无法被回收，而在Java7中subString方法会直接新建字符串。
  11. java运行时区域：线程不共享的区域：程序计数器（当前线程指令运行到的地址）、虚拟机栈（java方法的栈，每一次方法的调用都是一个栈帧的出栈和入栈，每一个栈帧包括了局部变量表、操作数栈、动态链接、方法出口）、本地方法栈（native方法的栈）；线程共享的区域：堆（JVM启动时创建的对象和数组，真实的数据内容）、方法区（存储了每一个类的结构信息，运行时常量池，字段和方法数据、构造函数等等，运行时常量池里面主要是编译器可知的数值字面量和在运行期才能解析获得的方法或字段的引用）
  12. JDK7之后，所有需要关闭的资源只要实现了java.lang.AutoCloseable接口，程序结束之后资源就会自动关闭，在try-with-resource语句中这样使用才会有效果，在try（）里定义多个资源，在结束之后是按照在try里的定义顺序自动关闭。
  13. JDK8里面通过lambda表达式以及函数式接口提供了很多排序手段，JDK8之前需要自己定义实现了Comparator接口的类，jdk8中可以利用list.sort函数添加，还可以利用Collections.sort函数（参数是list和Comparator提供的静态方法）
  14. lambda表达式：（parameters）->{表达式}，lambda语句可以指定参数类型和return，也可以不指定，编译器会自动推断，后面的表达式如果只有一句就可以不用加大括号了。参数的括号不能省略。
  15. lambda表达式专门针对只有一个方法的接口（函数式接口），使用@FunctionalInterface可以构造自己的函数式接口。jdk提供了常用的函数式接口，Function：提供任意一种类型的参数，返回另外一个任意类型返回值。 R apply(T t); Consumer：提供任意一种类型的参数，返回空值。 void accept(T t); Supplier：参数为空，得到任意一种类型的返回值。T get(); Predicate：提供任意一种类型的参数，返回boolean返回值。boolean test(T t); BinaryOperator：提供两个参数，返回一个参数。当lambda表达式想要代替这些函数式接口的时候返回的类型和参数应该满足这些函数式接口的方法签名。函数式接口作为参数会在特定的类或者方法中被调用接口的方法，这时候可以直接使用lambda表达式替代传入的函数式接口。lambda表达式内部的this指的是包含lambda表达式的类，使用内部匿名类的方法this指的是匿名内部类本身。
  16. 方法引用：只要是方法的返回值类型和参数与函数式接口中抽象方法的参数和返回值类型保持一致就可以使用方法引用代替函数式方法，方法引用有三种方式：构造器方法引用(Class::new)、类普通方法引用(Class::method)、类静态方法引用(Class::static_method)、实例方法引用(instance::method)，System.out::println就是实例方法引用。
  17. lambda表达式在编译阶段会被转换为匿名内部类，自由变量是在lambda表达式之外定义的变量，自由变量在lambda表达式内部不能被改变。
  18. Stream：不存储数据、不改变原数据同时延迟执行，这里的stream和IO流不一样，这里的Stream是对数组和集合的操作流，对于stream的聚合、消费或收集操作只能进行一次，创建流的方式Arrays.stream、集合.stream、集合.parallelStream、Stream.empty、Stream.generate().limit()，map函数可以把一种类型的流转换为另一种类型。
  19. Optional类型：Optional表示一个安全的指定结果类型，所谓的安全指的是避免直接调用返回类型的null值而造成空指针异常，调用optional.ifPresent()可以判断返回值是否为空，或者直接调用ifPresent(Consumer<? super T> consumer)在结果部位空时进行消费操作，Optional创建的静态方法of和ofNullable(可以接受空值)；
  20. 接口的默认方法和静态方法：接口的默认方法的应用场景是当接口中新增了方法声明之后，所有实现了该接口的方法都需要重新实现该方法，而接口的默认方法则是可以让接口的实现类不需要重写方法也可以执行该函数，静态方法是可以直接被调用的，不需要接口的实现类，default和static不能同时使用，当一个类实现的多个接口出现同样的默认方法的时候，这个类必须覆写这个方法，不然会报错，还有就是接口的方法的优先级：子接口的优先级高，类实现的同名方法的优先级高。

  1. 字符串的不可变性：之前的字符串对象不会销毁，而是会在堆中重新新建字符串对象并指向。

  3. java中的异常主要分为四类：IOException、ClassNotFoundException、CloneNotSupportedException、RuntimeException（唯一的运行时异常），前三种异常必须在编译之前就捕捉或者抛出。

  6. 序列化之后属性有变化会导致值的丢失；父类没有序列化，但是子类序列化了，子类的属性可以序列化，但是父类的属性会丢失；不想序列化的字段加上transient即可。
  7. 正整数的原码补码反码都是一样的，负整数的原码首位为1，反码是符号位不变，其余相反，补码是符号位不变，反码加1
  8. Set中的元素是无序且不重复的，这句话中的Set指的是HashSet，LinkedHashSet是按照添加的顺序排列，而TreeSet是按照自然顺序a~z
  9. URLClassLoader是ClassLoader的子类，URLClassLoader可以加载任意路径下的类，ClassLoader只能加载ClassPath下的类
  10. long类型不同进制的转换是从左到右依次遍历每个字符乘以进制数，再加上下一个字符，接着再乘以进制数，不断重复。
  11. 集合遍历是使用Iterator, Iterator是工作在一个独立的线程中，并且拥有一个互斥锁。Iterator 被创建之后会建立一个指向原来对象的单链索引表，当原来的对象数量发生变化时，这个索引表的内容不会同步改变，所以当索引指针往后移动的时候就找不到要迭代的对象，所以按照 fail-fast原则 Iterator 会马上抛出java.util.ConcurrentModificationException 异常。所以 Iterator 在工作的时候是不允许被迭代的对象被改变的
  12. 装箱过程是通过调用包装器的valueOf方法实现的，而拆箱过程是通过调用包装器的 xxxValue方法实现的。（xxx代表对应的基本数据类型）
  13. “==”运算符既可以用来比较基本类型变量和引用类型变量。当两个操作数都是包装器类型的变量时，判定标准为他们是否指向同一个对象；而如果其中有一个操作数是表达式（即包含算术运算）则会先进行自动拆箱，再进行对应基本类型变量比较。
  14. 静态变量被所有的对象所共享，在内存中只有一个副本，它当且仅当在类初次加载时会被初始化。而非静态变量是对象所拥有的，在创建对象的时候被初始化，存在多个副本，各个对象拥有的副本互不影响。
  15. 先加载父类，然后再加载子类，先调用父类的构造器，在调用自身的构造器。
  16. static关键字有一个比较关键的作用就是 用来形成静态代码块以优化程序性能。static块可以置于类中的任何地方，类中可以有多个static块。在类初次被加载的时候，会按照static块的顺序来执行每个static块，并且只会执行一次。
  17. ArrayList底层是数组，LinkedList底层是链表，Vector底层也是数组，Vector支持线程同步，前两个不支持，链表的查找速度较慢，因为需要移动指针，数组的查找速度较快（索引），数组的增删速度比较慢，因为需要移动后面的内容，这时候链表就比较快。stack是Vector的子类，主要实现了后进先出的栈的功能。当需要扩展数组长度的时候，Vector会扩展为原来的一倍，ArrayList会扩展50%，
  18. SynchronizedList和Vector的区别：SynchronizedList是由Collections静态工具类提供的一个可以实现线程同步的list，和Vector的区别在于以下几点：Vector和SynchronizedList在长度扩展上的区别；Vector是通过同步方法实现同步，而SynchronizedList是通过同步代码块的方式实现（同步代码块内部还是list的方法），SynchronizedList可以通过传入的参数指定锁定的对象，Vector只能锁定this；使用SynchronizedList的时候，进行遍历需要手动同步处理；SynchronizedList可以将list的所有子类转换为线程安全的类
  19. HashTable和HashMap以及ConcurrentHashMap的区别：hashtable可以实现同步；HashMap支持插入null值和null key，原因在于HashMap会将null的hashcode定为0，底层都是Entry数组的实现方式，Entry四个属性：Key、Value、hash、Entry（指向下一个Entry，出现hash碰撞）；HashMap的大小总是2的幂次，所以说每次扩容都变成原来的两倍，通过位运算实现除法，提高效率；ConcurrentHashMap允许多个修改操作并发进行，锁定Map中的某个段，而HashTable 是锁定整个Map的。对于HashMap来说jdk1.7使用数组加链表（Entry），jdk1.8使用数组加链表和红黑树实现（Node）
  20. Java7提供了一种并行任务框架，Fork/Join框架 ，ForkJoinPool是框架的任务调度器，提供了三个调度子任务的方法：execute（没有返回结果），invoke、invokeAll（有返回结果），submit（返回future对象），具体的任务执行类需要实现RecursiveAction（无结果返回）和RecursiveTask（有结果返回），任务执行类里面有一个compute函数，类似于递归的思想，这个compute判断是否可以计算，如果可以计算就直接计算，不能计算就进行拆分，fork就是执行execute，join函数就是得到返回值。
  21. 并发就是在一段时间内多个任务在执行，交替或者串行或者其他，但是并行就是指并行执行，多个任务在同时执行，
  22. java实现多线程的三种方式：继承Thread类；实现Runnable接口，重写run方法（一个子类只能继承一个父类），将Runnable实例传给Thread对象；实现Callable接口，重写call方法，将Callable实例传给FutureTask对象，再传给Thread对象
  23. Futuretask是实现了RunnableFuture接口的类，RunnableFuture接口继承了Runnable接口和Future接口，所以FutureTask的使用方法有两种：FutureTask+ExecuteService和FutureTask+Thread，第一种方法就是把实现了Callable接口的类传递给FutureTask，第二种方法就是把FutureTask传给Thread，并利用FutureTask获得结果。
  24. Join方法的主要作用是挂起当前线程，等到调用了Join方法的线程结束之后，当前线程才会继续运行。
  25. 线程的状态一共有6个，NEW是刚创建的线程，还没有启动；RUNNABLE是正在运行的线程；TIME_WAITING是有时限的等待线程；而WAITING是无限的等待，等待一个特殊事件的唤醒；BLOCKED表明这个线程当前处于等待锁的阻塞状态，一旦获得锁将进入RUNNABLE状态；TERMINATED是指线程运行结束，进入终止状态。
  26. 死锁：多个线程拿了对方需要的锁，同时又在等待对方释放自己需要的锁；活锁就是多线程相互谦让，导致没有线程正常执行；饥饿就是说低优先级一直抢不过高优先级的线程，导致低优先级的线程得不到执行；无锁就是对资源不锁定，一堆线程抢着修改共享资源。
  27. 避免死锁的方法：按顺序加锁（这样能保证只能有一个线程获得锁）；获取锁时限（一定时间之后放弃自己已经拥有的锁）；检测死锁（出现死锁就采取策略）
  28. 线程池核心类：ThreadPoolExecutor类的主要参数是corePoolSize（核心线程池的大小。线程池的最小线程数量）、maximumPoolSize（线程池的最大线程数量），KeepAliveTime（超过corePoolSize的线程空闲超过KeepAliveTime就会被销毁），unit（KeepAliveTime这个时间的单位），WorkQueue(存储等待执行的工作队列)，ThreadFactory（创建线程的工厂），handler（拒绝策略，默认抛出异常）；线程池的工作流程：如果线程池中的线程数量小于corePoolSize的话，就会直接创建新线程执行任务，如果已经超过corePoolSize的话就会存到workQueue，workQueue存满之后，当线程数小于maximumPoolSize的时候就会新建线程执行任务，当线程数超过maximumPoolSize就会利用handler执行拒绝策略。核心类ThreadPoolExecutor有四个在Executor下的静态实现类：newFixThreadPool（固定大小的线程池，核心和最大线程数相等），newCacheThreadPool（带缓冲的线程池，核心线程池大小为0，有新任务就会执行），newSingleThreadPool（单线程线程池，核心和最大线程数为1，每次保证有1个线程在执行）、newSchedulerThreadPool（调度线程池，按照一定的周期执行任务），拒绝策略：AbortPolicy（直接抛出拒绝异常）、CallerRunsPolicy（让调用者线程执行新任务）、DiscardPolicy（直接丢弃）、DiscardOldestPolicy（从队列中丢弃掉最老的任务），提交线程使用execute和submit，不同点在于submit会返回一个future对象，关闭线程池使用shutdown（执行完所有任务再关闭）和shutdownNow。
